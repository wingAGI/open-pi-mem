MEM 复现相关开源数据调研

主题：适合构造 MEM 的高层/低层训练数据的数据集
生成日期：2026-03-06 官方文档核对
重点问题：是否有 subtask 标注、是否能拿到 success/failure、原始数据字段长什么样
结论导向：为 open-pi-mem 选高层与低层数据路线

一句话结论

最适合高层 MEM 标注

Isaac Lab Mimic / SkillGen 最强，因为它明确围绕 subtask boundary、subtask_term_signal、success flag 组织数据与生成流程。

subtask: 强 success/failure: 强 机器人: 仿真

最适合低层最小实验

DROID 和 BridgeData V2 最现实，因为有真实机器人图像、动作和语言 instruction，但它们通常没有现成的 episode 内 subtask/success-failure，需要你额外构造。

subtask: 弱到中 success/failure: 弱 机器人: 真实

总览对比

数据集 类型 subtask 可得性 success/failure 可得性 是否有真实机器人动作 更适合 MEM 哪一层 结论
Isaac Lab Mimic 仿真机器人 强：环境直接围绕 subtask 配置与边界注释设计 强：有布尔 success criteria 和每个 subtask 的 term signal 有 高层优先，也能做低层 最适合 MEM 风格数据管线原型
Isaac Lab SkillGen 仿真机器人 强：官方给预标注数据，要求 start/term boundaries 强：生成时设置 episode success flag，外层过滤成功样本 有 高层最佳 如果目标是最干净的 `[历史子任务 + 成败]`，这是首选
DROID 真实机器人 中：有 step-level language instruction，但不是显式 subtask segments 弱：有 terminal/reward，但没有子任务级 success 标注 有 低层最佳；高层需你额外构造 最适合真实机器人 low-level 起步
BridgeData V2 真实机器人 弱：每条轨迹只有自然语言 instruction 弱：官方站点不提供子任务 success/failure 有 低层 适合作为 low-level BC 数据，不适合作为现成高层标注源
Ego-Exo4D Keystep 第一视角人类技能视频 强：明确的 segment + step_name + step_description 中弱：有 is_essential，但不是 success/failure 无 高层 适合做 step/history/memory 建模，不适合低层机器人动作
Ego4D Moments / Narrations 第一视角长视频 中：有带起止时间的 labels 和 summary_text 弱：没有任务式 success/failure 无 高层辅助预训练 可用于长时语义压缩，但离 MEM 的机器人高层 supervision 还有距离
EPIC-KITCHENS 第一视角厨房视频 中：有大量 action segments 和 narrations 弱：没有显式子任务成败 无 高层辅助预训练 适合动作段与叙述建模，不适合直接构造 MEM 的成败历史

推荐路线

路线 A：最稳的 MEM 复现路线

高层使用 Isaac Lab Mimic / SkillGen 构造 [历史子任务 + 成败] → [next_subtask, next_memory]；低层使用 DROID 或 BridgeData V2 做真实机器人动作建模。

路线 B：先做纯高层原型

高层使用 Ego-Exo4D Keystep 或 Ego4D/EPIC 先验证 memory summary 生成流程，再用少量人工或规则补 success/failure。

数据集逐项说明

1. Isaac Lab Mimic

定位：Isaac Lab 的 imitation/data generation 工作流。它不是一个单独的“大而全下载数据集”页面，而是一套环境、标注、生成 demo 的机制。对 MEM 来说，它最大的价值是：官方定义了 subtask、boundary、term signal、success criteria 这些概念本身。

官方证据：Mimic 文档写明，生成 candidate demonstrations 时使用 boolean success criteria 判断是否成功；成功才加入输出数据集。还要求环境实现 get_subtask_term_signals，返回每个 subtask 是否完成的二值标志。

subtasks: list[SubTaskConfig]
get_subtask_term_signals() -> dict[subtask_name, bool]
boolean success criteria -> decide whether episode succeeded

你能直接拿来干什么：

  • 直接构造 episode 内 subtask boundary
  • 直接得到 per-subtask completion signal
  • 直接构造 success/failure supervision

真实样例：官方文档没有直接贴出一条完整 demo row，但明确给了真实任务例子 FrankaCubeStackIKRelMimicEnvCfg，对应的是 Franka 立方体堆叠任务；并要求环境返回类似下面的子任务完成信号：

{
  get_subtask_term_signals() -> {
    "grasp_cube": bool,
    "stack_cube": bool,
    ...
  }
}

官方还明确说明，candidate demonstration 会经过 boolean success criteria 判定是否成功，成功才写入输出数据集。

适配 MEM 结论：如果你要一个最干净的高层监督来源，它比 DROID/Bridge 更合适。

来源：Isaac Lab Mimic 文档

2. Isaac Lab SkillGen

定位：在 Mimic 基础上，专门围绕技能片段重组和生成设计。官方直接提供一个 annotated_dataset_skillgen.hdf5 示例数据，并明确说包含 manually annotated subtask boundaries。

具体格式/机制：

  • 源数据文件：dataset_skillgen.hdf5
  • 标注后数据文件：annotated_dataset_skillgen.hdf5
  • subtask 定义依赖 object_ref、subtask_term_signal、subtask_start_signal
  • 边界由二值信号 0→1 的跳变定位成 [t_start, t_end]
  • 数据生成阶段会 set the episode success flag，外层流程过滤和消费成功 episode
{
  subtask_start_signal: binary 0/1,
  subtask_term_signal: binary 0/1,
  subtask boundaries: [t_start, t_end],
  episode success flag: boolean
}

真实样例：官方文档虽然没有直接贴出 HDF5 内单条 JSON row，但给了真实任务级样例和信号定义。SkillGen 文档举的技能片段例子是“抓取一个 cube 时，start signal 在 gripper 闭合前，termination signal 在物体被抓起后”；相关任务是 cube stacking。官方还给了预标注文件名 annotated_dataset_skillgen.hdf5。

{
  subtask_start_signal: 0 -> 1,
  subtask_term_signal: 0 -> 1,
  subtask boundary: [t_start, t_end],
  episode success flag: bool
}

相关教程和 MimicGen/SkillGen 任务示例里，真实子任务序列类似：

1. grasp cubeA
2. place cubeA on cubeB
3. grasp cubeC
4. place cubeC on cubeA

适配 MEM 结论：这是目前最接近你想要的 `[历史子任务 + 成败]` 的官方开源来源。

来源：SkillGen 官方文档

3. DROID

定位：大规模真实机器人操作数据集，适合 low-level action learning。官方 RLDS schema 写得很具体。

RLDS 关键字段：

{
  steps: {
    is_first, is_last, is_terminal,
    language_instruction,
    language_instruction_2,
    language_instruction_3,
    observation: {
      gripper_position,
      cartesian_position,
      joint_position,
      wrist_image_left,
      exterior_image_1_left,
      exterior_image_2_left,
    },
    action_dict: {...},
    reward,
    action
  }
}

优点：

  • 有真实机器人图像、动作、状态
  • 有 step-level language instruction
  • 有 RLDS 和 raw data 两种形式
  • 官方还提供 droid_100 作为 100 个 episode 的调试版本

缺点：

  • 没有显式 episode 内 subtask segment 标注
  • 没有 per-subtask success/failure 标注
  • 虽然有 reward 和 is_terminal，但更接近 episode 级或 final-step 级信号，不足以直接还原高层历史

真实样例：官方 schema 直接给了 RLDS episode 结构，是真实发布格式的一部分：

{
  "steps": {
    "is_terminal": true,
    "language_instruction": "...",
    "observation": {
      "gripper_position": ...,
      "cartesian_position": ...,
      "joint_position": ...,
      "wrist_image_left": ...,
      "exterior_image_1_left": ...
    },
    "reward": 1.0,
    "action": [6x joint velocities, 1x gripper position]
  }
}

文档还给了真实可下载调试集：gs://gresearch/robotics/droid_100，这是 100 个 episode 的 RLDS 样例版本。

适配 MEM 结论：非常适合 low-level；高层必须你自己从 instruction、state、reward、retry 中额外构造。

来源：DROID Docs、DROID 项目页

4. BridgeData V2

定位：真实机器人多任务 manipulation 数据。官方项目页说明每条 trajectory 有自然语言标注。

已知官方信息：

  • 53,896 trajectories
  • 13 skills
  • 24 environments
  • 每条轨迹有一条 natural language instruction
  • 控制频率 5Hz，平均 38 steps
  • 相机为一个固定 RGBD + 两个随机位姿 RGB，相片 640×480
{
  trajectory-level language instruction,
  image observations,
  robot actions,
  avg length: 38 steps,
  5Hz teleop demos
}

缺点：官方页面没有给 episode 内 subtask segments，也没有给 per-subtask success/failure。

真实样例：官方网站的随机轨迹浏览器会展示真实轨迹对应的自然语言标注；页面上公开展示的语言指令例子包括：

put the carrot on the plate
flip the pot upright
put the mushroom in the pot
move the cloth to the left

官方说明这些 instruction 是 trajectory-level 的 natural language annotation，而不是 episode 内多个 subtask。

适配 MEM 结论：低层很好；高层如果要用，只能自己切 subtask 并补 success/failure。

来源：BridgeData V2 项目页

5. Ego-Exo4D Keystep

定位：第一视角/第三视角技能视频数据里的关键步骤标注。它不是机器人动作数据，但很像“高层子任务序列”。

JSON 关键字段：

{
  annotations: {
    take_uid,
    scenario,
    segments: [
      {
        start_time,
        end_time,
        step_id,
        step_unique_id,
        step_name,
        step_description,
        is_essential
      }
    ]
  },
  taxonomy,
  vocabulary
}

优点：

  • 天然有 segment 边界
  • 天然有 step 名称和 step 描述
  • 还有 taxonomy，可做层级步骤建模

缺点：

  • 没有机器人 action
  • is_essential 不是 success/failure
  • 更适合高层语义，不适合低层控制

真实样例：官方直接贴了 sample JSON，真实例子来自 Covid-19 Rapid Antigen Test 场景：

{
  "take_uid": "77cd56fc-6b68-4c7f-814b-81545484dd2d",
  "take_name": "georgiatech_covid_07_6",
  "scenario": "Covid-19 Rapid Antigen Test",
  "segments": [
    {
      "start_time": 0.85598,
      "end_time": 33.20377,
      "step_id": 31,
      "step_name": "Unbox package",
      "step_description": "Unbox covid19 test package",
      "is_essential": true
    }
  ]
}

适配 MEM 结论：如果你要做 high-level memory summary pipeline 原型，它非常好；但别把它当真实机器人高层标注的等价替代。

来源：Ego-Exo4D Keystep

6. Ego4D Moments / Narrations

定位：大规模第一视角长视频及其时间段事件标注、narration 和 summary。非常适合“长时语义压缩”研究。

Schema 关键字段：

{
  moments_.json: {
    videos[].clips[].annotations[].labels[]: {
      start_time, end_time, label, primary
    }
  },
  narrations.json: {
    narration_pass_1/2: {
      narrations[]: { timestamp_sec, narration_text },
      summaries[]: { start_sec, end_sec, summary_text }
    }
  }
}

优点：

  • 有时间段 labels
  • 有 narration 和 summary_text
  • 天然适合训练“历史压缩成文本”

缺点：

  • 不是机器人
  • 没有任务式 success/failure
  • 只能作为高层辅助，不适合直接做 MEM 全流程监督

真实样例：官方 schema 中的字段例子包括：

{
  "labels": [{
    "start_time": 12.3,
    "end_time": 18.7,
    "label": "open refrigerator",
    "primary": true
  }],
  "narrations": [{
    "timestamp_sec": 15.2,
    "narration_text": "I open the fridge"
  }],
  "summaries": [{
    "start_sec": 0.0,
    "end_sec": 60.0,
    "summary_text": "preparing ingredients in the kitchen"
  }]
}

来源：Ego4D Annotation Schemas

7. EPIC-KITCHENS

定位：第一视角厨房视频数据。官方强调有大量 action segments 和 narrations。

官方概况：

  • 100 hours
  • 20M frames
  • 90K action segments
  • 20K unique narrations
  • 97 verbs / 300 nouns

监督形式：

  • trimmed action segments
  • (verb, noun) labels
  • caption / narration 对应 action segments

缺点：

  • 没有显式 success/failure
  • action segment 更像动作识别片段，不一定等于任务子目标
  • 不适合直接当 MEM 的机器人高层监督

真实样例：官方注释仓库直接给了字段和示例值，例如 action annotation CSV 里有：

narration_id: P01_01_0
start_timestamp: 00:00:00.14
stop_timestamp: 00:00:03.37
narration: open door
verb: open
noun: door

另外还有类似：

narration: put lunch box
narration: turn on tap
narration: take plate

来源：EPIC-KITCHENS 2024、EPIC-KITCHENS-100 annotations

对 open-pi-mem 的直接建议

高层脚本先接谁

先接 Isaac Lab SkillGen/Mimic 风格数据。原因：你需要的是 [历史子任务 + 成败]，而不是只有一条 instruction。这里的字段最接近你真正需要的高层 supervision。

低层脚本先接谁

先接 DROID。原因：它的 RLDS schema 明确、图像和动作字段齐全，还有 droid_100 调试版本，最适合先验证 low-level data adapter。

如果你暂时拿不到 Isaac Lab 数据

可先用 Ego-Exo4D Keystep 做高层 history/summary 原型，但 success/failure 需要你自己补，最多当弱监督。

不要误判的地方

DROID/Bridge 的 language instruction 不等于 subtask 标注。 这两者经常只有 trajectory-level 或 step-level instruction，而不是带边界和成败的高层历史。

证据出处

  • Isaac Lab Mimic：subtask annotation、heuristics、term signals、boolean success criteria
  • Isaac Lab SkillGen：pre-annotated dataset、manual subtask boundaries、start/term signal、episode success flag
  • DROID Docs：RLDS schema、language_instruction、reward、action、observation
  • DROID 项目页：TFDS quickstart
  • BridgeData V2：trajectory-level natural language instruction、5Hz、38 timesteps
  • Ego-Exo4D Keystep：segments、step_name、step_description、is_essential
  • Ego4D Annotation Schemas：moments labels、narrations、summaries
  • EPIC-KITCHENS 2024：90K action segments、20K narrations
这里有一部分结论属于“基于官方 schema 的工程推断”：例如 DROID 和 BridgeData V2 不适合作为现成高层 supervision 源，不是因为它们完全不能做，而是因为官方公开格式里没有直接给出 MEM 需要的子任务边界和子任务成败标签。
